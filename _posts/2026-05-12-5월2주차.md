# A.
---
## 1. SRP 와 OCP 
---
### 1.1. SRP
---
단일 책임 원칙은 **"하나의 객체는 하나의 명료한 기능(책임)을 가져야 한다."** 라는 원칙이다.

이는 프로그램의 유지 보수와 관련이 있는 원칙이다.

하나의 기능이란 말은 **프로그램 안에서 담당하는 기능적 부분**을 말한다.  프로그램의 설계에 따라 이 기능이 어떤 것을 담을지는 설계자의 자유이지만, 객체의 기능이 다른 여러 프로그램에서 쓰일수록 해당 객체의 변경시 미쳐지는 파급효과는 더욱 커진다.

예시는 다음과 같다.

```java
class CustomClass {
	...
	public String GetDate(){
		...
		return date // return "YYYY-mm-dd HH:MM:SS" format datestring 
	}
	...
}

class UseClass {
	...
	public String Year(){
		String year = GetDate.subString(0,4);
		... // year 을 사용하는 로직
	}
	...
}

class UseClass2 {
	...
	public String Hours(){
		String hours = GetDate.subString(11,13);
		... // hours 을 사용하는 로직
	}
	...
}
```

이 예시에서 `GetDate()`를 살펴보자.

`GetDate()`매서드는 완성된 타임 스탬프 문자열을 반환하는 기능을 가지고 있다. 이 매서드 자체로는 큰 문제가 없다. 
그러나, `UserClass` 들이 이를 사용할 때, 문자열을 잘라서 사용하는 것이 문제가 된다.

이 예제에서는 **완성된 타임스탬프** 가 하나의 기능이 아니라, `year`, `hours` 와 같은 **부분적인 문자열이 하나의 기능** 을 담당하게 된다. 
즉, `GetDate()` 매서드는 
- `year` 을 반환하는 기능
- `hours` 를 반환하는 기능
두 가지 기능을 담당하게 되는 것이다. 이런 경우가 단일 책임 원칙의 위반이다.

> 만약, date 의 반환 포맷이 달라지면, **`UserClass`들의 함수들도 전부 바꾸어야 한다.**

그래서 `GetDate()` 함수를 `GetYear()` 과 `GetHours()` 처럼 작은 단위로 나누고, `GetDate()` 함수를 각 부분적인 스탬프를 모아서 반환하는 함수로 만들면 각 기능마다 객체를 할당한, 단일 책임 원칙을 충족한 코드가 된다.

```java
class CustomClass {
	...
	public String GetDate(){
		...
		return fullFormatDate // return "YYYY-mm-dd HH:MM:SS" format datestring 
	}
	
	public String GetYear(){
		...
		return years // return "YYYY" format datestring 
	}
	
	public String GetHours(){
		...
		return hours // return "HH" format datestring 
	}
	...
}

class UseClass {
	...
	public String Year(){
		String year = GetYear()
		... // year 을 사용하는 로직
	}
	...
}

class UseClass2 {
	...
	public String Hours(){
		String hours = GetHours()
		... // hours 을 사용하는 로직
	}
	...
}
```

클래스의 경우도 비슷하다.

```java
//before
class UsedClass {
	...
	public void method(){
		// 이 메서드가 변하면
		...
	}
	
	public void methodForFn1(){
		...
		this.method()... // 이부분을 고치고
	}
	
	public void methodForFn2(){
		...
		this.method()... // 이부분도 고쳐야 한다.
	}
	
	...
}
```

- `fn1`
- `fn2`
의 기능을 담당하는 UsedClass 를

```java
//after
class UsedClass {
	public void methodForFn1(){ // 이 기능을 고친다면
		...
			ClassForFn1.Fn1() 
		...
	}
	public void methodForFn2(){
		...
			ClassForFn2.Fn2()
		...
	}
}

class ClassForFn1 {
	public void Fn1(){...} // 이 부분만 고치면 된다.
}


class ClassForFn2 {
	public void Fn2(){...}
}
```

와 같이, 
- `fn1` => `ClassForFn1`
- `fn2` => `ClassForFn2`
- 메서드를 제공하는 유저 인터페이스 => `UsedMethod`
로 하게되면 각각의 기능이 단일한 클래스를 가짐으로 SCP 를 지키는 코드가 되는걸 알 수 있다.
### 1.2. OCP
---
OCP 는 **확장에는 개방적으로, 변경에는 폐쇄적으로 작성된 로직** 이라는 원칙이다.

즉, 변경사항이 들어왔을때
- 종래의 객체를 수정하지 않는다(변경에는 폐쇄적)
- 새로운 객체를 생성해서 변경을 커버한다(확장에는 개방적)
으로 **추상화, 다형성, 확장을 적극적으로 사용** 하라는 것 이다.

바로 예를 들어보자. 
폴더의 파일을 관리하는 파일 시스템 매서드를 작성한다 생각해본다.

```java
class OfficeFileHandler {
	...
	public OfficeFile useFileFn(String path){...}
}

class OfficeFile{
	private String path;
	...
	public String getExcelFileToJson(){...}
	public String getWordFileToJson(){...}
	// if you want to convert text file to json data.
	// add new method.
	public String getTextFileToJson(){...}
	...
}
```

위 예시의 `OfficeFile` 은 JSON 형식 데이터로 반환하는 매서드를 가지고 있는데, XML 이나 YAML, CVS 형식의 데이터로 받고 싶으면 
1. 매서드를 추가하거나
2. 별도의 클래스를 만들거나
두가지의 방법이 있다.

그러나 두가지의 방법 모두 각각의 문제점이 있는데,
1. 매서드를 추가할 시, **`OfficeFile` 클래스의 변경**이 된다.
2. 클래스를 추가할 시,  원래 클래스의 변경은 없지만, **새로운 함수(ex - `getExcelFileToCVS()`)를 사용하는 코드를 작성**해야 한다.

때문에 기존의 코드를 최대한 변경 시키지 않기 위해 다형성과, 확장을 사용하는 것 이다.

```java
class OfficeFileHandler {
	...
	public OfficeFile useFileFn(String path){...}
}

interface OfficeFile{
	...
	public String getExcelFile(String path);
	public String getWordFile(String path);
	public String getTextFile(String path);
	...
}

class ToJson implaments OfficeFile {
	@Override
	public String getExcelFile(String path){...}
	...
}
class ToCVS implaments OfficeFile {
	@Override
	public String getExcelFile(String path){...}
	...
}
```

이렇게 작성할 경우, 객체 생성시의 클래스 변경 만으로 반환 되는 데이터의 형식을 바꿀 수 있다. 
또한, Yaml 형식을 추가한다고 해도 `ToYaml` 과 같은 클래스를 추가하기만 하면 된다.

기존 클래스의 변경은 적으며, 새로운 기능의 추가는 확장적으로 쉽게 처리할 수 있다.

## 2. StreamAPI 와 map / flatMap
---
`map()`과 `flatMap()` 의 가장 큰 차이는 **컬렉션의 순회 깊이를 얼마나 들어가느냐** 라고 생각한다.

바로 예시를 들어보자.
```java
List<int> intArr = List.of(1,2,3,4,5);
List<Array[int]> arrArr = List.of([1],[2],[3],[4],[5,0]);

intArr.stream()
	.map(n -> n > 3)
	.forEach(System.out::println) // 3 4 5


arrArr.stream()
	.map(n -> n > 3) // exception! - type of n == Array
	.forEach(System.out::println) 


arrArr.stream()
	.flatMap(n -> n > 3) // like [1,2,3,4,5,0] 
	.forEach(System.out::println) // 3 4 5
```

예시에서 본 것과 같이, map 은 컬렉션의 내부 요소를 그대로 확인한다. 이는 단점이자 장점인데,

- 단점
	- 요소 내부가 원시 자료형이 아닐 경우, 각 요소마다 사용할 접근 로직을 한번 더 짜줘야 한다.
- 장점
	- 컬렉션의 **구조가 유지된다**.

`map()`장점에서 말한 것 처럼 `flatMap()`은 원래 **객체의 구조가 "1단계 평탄화"** 된다. 때문에 원래의 객체가 어떤 구조 였는지 결과물을 가지곤 알 수가 없다.

`flatMap()`실습
```java
List<List<List<Integer>>> test =  // [[[1,2],[3,4,5]],[[6,7],[9,10]]]
        List.of(  
                List.of(  
                        List.of(1, 2), List.of(3, 4, 5)  
                ),  
                List.of(  
                        List.of(6,7), List.of(9, 10)  
                )  
        );  
test.stream()  
        .flatMap(n -> n.stream())  
        .forEach(System.out::println);
        
// result - 깊이 1까지 직렬화 됨.
//[1, 2]
//[3, 4, 5]
//[6, 7]
//[9, 10]
```
