# A. git rebase 와 merge 의 차이
---
merge 는 **두 브랜치를 하나로 합쳐서 하나의 브랜치로 만드는 것** 이고,  rebase 는 **현재 브랜치를 타겟 브랜치에서 뻗어나온 브랜치로 만드는 것** 이다.

때문에 과정과 결과에 소소한 차이가 있는데, 그 항목은 다음과 같다.

- conflict 처리 
	- merge : 브랜치의 최종 커밋끼리 생긴 충돌을 해결해 주어야 함.
	- rebase : 베이스가 될 **브랜치의 모든 커밋에 대한 conflict** 가 해결 되어야 함.
- 결과
	- merge : 단일 브랜치의 최종 커밋
	- rebase :  베이스 브랜치의 최종 커밋과 그곳에서 분기한 현재 브랜치


## ex) git rabase 실습
---
### 개요
---
브랜치는 다음과 같다

- main
	- init data add
- dev
	- dev data add 
- test
	- test data add 
	- test2 data add

dev 를 test 브랜치로 리베이스 할 때, 어떤 일들이 진행되는가를 실습해서 적어놓았다. 


```bash
# all log status
$ git log --all --graph --oneline
* 0d0e914 (test) test data 2 added.
* 6431236 test data added.
| * 746c16c (dev) dev data added.
|/
* 4088d7c (main) initiation
```

### 예상되는 문제
---
dev 와 test 브랜치의 내용에 공통 데이터가 아닌 부분이 있어서 confilict 가 생길 것이다.

>conflict 처리
>
>dev 의 커밋들을 차례차례 적용하면서 진행한다.
>즉 dev 의 커밋이 test 의 커밋들과 컨플릭트가 생기지 않을 때 까지 해야한다.
>그리고 위 과정을 모든 dev 의 커밋에 적용될때까지 한다.

### 실제
---
실제 conflict 이후, `--continue | --abort  | --skip` 을 옵션으로 넣어서 진행 할 수 있다.

- `--continue` : 충돌 해결 후, 리베이스 계속 진행
- `--abort` : 리베이스를 중지하고, 리베이스 이전상태로 복귀
- `--skip` : 현재 커밋을 건너뛰고 진행


충돌 처리 후 실제 진행된 로그들은 다음과 같다

```bash
# run command
$ git rebase test
$ git log --all --oneline --graph
* 25b8893 (HEAD -> dev) dev data added.
* 0d0e914 (test) test data 2 added.
* 6431236 test data added.
* 4088d7c (main) initiation
```

데이터의 내용은 다음과 같다.

```bash
# in dev
user@~/Local_Documents/_Codeit_Bootcamp/Code(dev)$ cat text1.txt
initiation state int text1

dev data text.

this is test data in this

this is test data 2

# in test
user@~/Local_Documents/_Codeit_Bootcamp/Code (test)$ cat text1.txt
initiation state int text1

this is test data in this

this is test data 2

```

즉 test 의 아래에 dev를 만든것 처럼 브랜치가  이동된다.


# A. git fetch 와 git pull 의 차이
---
Fetch 는 현재 내 폴더의 상태를 유지하면서 remote 의 파일을 쓰는 것. 
pull 은 remote 의 상태를 그대로 복원하는 것 이다.

>pull 을 할 시, 이미 내 작업 공간내의 파일이 존재하고, 그 파일이 원격 레포지토리와 다른 커밋을 가진다면, 에러를 뱉으면서 풀링이 취소된다.
