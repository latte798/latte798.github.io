# Q.
---
1. HashSet의 내부 동작 방식과 중복 제거 메커니즘을 설명하고, HashSet이 효율적인 중복 체크를 할 수 있는 이유를 설명해주세요.

2. O(n)과 O(log n)의 성능 차이를 실생활 예시를 들어 설명하고, 데이터의 크기가 1백만 개일 때 각각 대략 몇 번의 연산이 필요한지 비교해주세요.

# A.
---
## 1. HashSet
---
`HashaSet`은 `Set` 인터페이스를 준수하는 구현체 중의 하나이다.

내부적으로 `HashMap` 을 사용해서 `HashMap.values()` 를 반환하는 형식으로 객체를 반환한다. 

인덱스 테이블은 해당 **객체의 해시값을 키**로, **값은 객체 그 자체**를 쓴다.

즉, 구조적으로 해시값을 테이블 인덱스로 활용해서 **같은 해시값(`.hashCode()`)을 키값으로 가진다면 같은 주소에 참조**되는 형태인 것 이다.

이러한 구조 때문에, 
- 리스트 내부의 객체에 순서가 없다.
- **같은 해시값을 가진 객체(중복 객체)** 의 체크가 쉽다.
	- 리스트를 순회하는 대신, 해시값을 주소로 참조되는 객체가 있는지 체크하면 된다.


## 2. 시간복잡도
---
`O(n)` 의 경우는 데이터의 크기가 커지면 데이터의 갯수에 비례하게 커지는 로직을 뜻한다.
단일 차원 배열의 순회와 같은 작업이 해당된다. 
- 배열의 크기가 1개 커지면, 순회작업의 횟수(객체를 체크하는 횟수) 도 1개 늘어난다.

`O(log n)` 의 경우는 `N`개의 객체를 `log(2) N` 회만큼 연산하는 로직을 뜻한다.
예를들어, **1부터 10까지 사이의 임의의 숫자 하나를 이분법으로 맞추는 작업**( 이진트리의 특정 최종 자식 노드를 찾는 로직과 비슷) 이 해당된다.

```
ex)
고른 숫자 : 4

5인가? down (1,2,3,4)
2인가? up (3,4)
3인가? up (4)
=> 찾는 객체 확정

연산횟수 : 3
log(2) 10 => 3.3... 
```

데이터의 갯수가 100만개가 있다면, 
- `O(n)` - 100만 회
- `O(log n)` - `log 1000000 / log 2` = 약 20 회
가 연산 횟수가 된다.
